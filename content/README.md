# Benefits Hub content layer

This folder holds all **human-authored** wording for the Benefits Hub. Coverage
facts and every dollar amount come from the CMS/PBP pipeline (the per-plan JSON
files generated by `preprocess.py` in the CMS Benefit Data package) — these
files only control *how we say it*.

See `BENEFITS_HUB_DATA_RECONCILIATION.md` for the full architecture (in `01-architecture/` of the handoff package; `artifacts/duos-pwa/docs/` in the app repo).

## Where to add things

```
content/
  registry.json                     ← map a CMS contract number to a carrier pack
  base/<benefit>.json               ← generic wording for a benefit type (all plans)
  carriers/<carrier>/<benefit>.json ← carrier-specific: vendors, networks, CTAs, phones
  contracts/<H#>/<benefit>.json     ← only if one contract differs from the carrier default
  plans/<H#_plan>/<benefit>.json    ← only for a one-off plan-specific rider
```

Resolution order (most specific wins): `base` → `carriers` → `contracts` → `plans`.

**To onboard a new carrier:** add its contract number(s) to `registry.json`,
create `carriers/<name>/`, and fill in one file per benefit the carrier promotes
(vendors, phone numbers, networks) using their Summary of Benefits. That's it —
plans render with base wording + PBP numbers even before the carrier pack exists.

## Rules (enforced by the card compiler)

1. **No literal dollar amounts** in any field except approved `eoc_facts`.
   Numbers are written as slots — `{fitness.copay}`, `{eyewear.max_plan_benefit}` —
   and resolved against the plan's PBP JSON at build time.
2. `eoc_facts` entries require `"approved": true` **and** a `"citation"` to the
   EOC or Summary of Benefits page. Missing either fails the build.
3. If a slot can't resolve for a plan (benefit not covered or value not filed),
   that fact row is dropped for that plan — it never renders as $0.
4. **ONE wallet = ONE card.** CMS files the same shared allowance in up to three
   places (OTC 13b, SSBCI items 13i, a VBID b19b package, pooled dollar in the
   Section D combined group). The compiler consolidates them into a single
   wallet card and cross-references or absorbs the members — a member NEVER sees
   two or three cards restating the same allowance, and a dollar amount appears
   on exactly one card. Detection uses `pbpCategories` in `taxonomy.json`; the
   full algorithm is COMPILER_SPEC.md §4a. (Found on Humana H7617-076, where
   "Special benefit packages" + "Everyday living support" + the OTC card all
   described the one $225/month Healthy Options card.)
