2 · Attach SB & EOC

PDFs attach as citation sources for eoc_facts and side-by-side QA. The compiler never parses them — that's a spec non-goal (numbers come from PBP only).

3 · Author the carrier pack

What a carrier pack is: the human-authored wording layer for one carrier — one small JSON file per benefit card holding the things CMS data can't know: vendor names (TruHearing, SilverSneakers), vendor phone numbers & URLs (become the card's call/visit button), network names ("Humana Dental Medicare Network"), how-to notes, and approved EOC facts (the only place a typed dollar is allowed — each needs approved:true + a citation to the SB/EOC you attached in step 2).

What it can NOT do: create, remove, or re-price a benefit. Coverage and every number come from the plan's PBP data — a pack only decorates cards that already gate on. A plan with no pack still renders fully, with generic base wording. That's why one pack per carrier scales to all of that carrier's plans.

Pick a card key, edit its JSON, apply — the compile updates live and the hard rules run (a literal dollar outside eoc_facts fails the build). Export downloads the files for 03-content/carriers/<carrier>/.

4 · Admin-tool rows (optional, §4b)

What admin-tool rows are: the DUOS ops team already maintains an internal admin tool with one free-text row per covered benefit per plan — plain-language "how to use this benefit" instructions written by humans (which vendor to call, which website, what to bring). This panel ingests those rows so that content reaches the member cards without re-typing it.

How to use it: paste the rows for this plan as a JSON array of {"plan_id","benefit_name","description_markdown"}. Admin plan IDs like H7617076000 (contract+pbp+segment, no separators) crosswalk to pipeline IDs automatically. What the compiler does with each row: the text renders as a "How to use this benefit" box on the mapped card · phones/URLs found in the text become the card's CTA if it has none · free-text dollar amounts and SSBCI-specific language on the universal OTC card are flagged in the report for review (numbers stay PBP-owned) · a benefit name missing from admin-benefit-mapping.json fails the build — never silently dropped · the PBP gate still rules: a row can never make an uncovered benefit appear.
